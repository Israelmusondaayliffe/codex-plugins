# Evidence Surfaces by Domain

Choose proof that observes the real result through the interface its user or audience will rely on.

## Research

- Claim-to-source table
- Primary-source citation check
- Reproduction of a calculation or extraction
- Explicit separation of fact, inference, and unknown

Avoid using the number of sources as proof of truth.

## Writing and documents

- Reader can find the decision or next action
- Required sections and fields are present
- Claims trace to approved sources
- The rendered file opens and preserves its structure
- A reviewer can complete the intended task from the document

Avoid treating grammar checks as proof that the document does its job.

## Creative work

- The artifact matches the approved brief and reference set
- Required subjects, composition, duration, or dimensions are present
- Visual or audio inspection checks the delivered file itself
- Variations preserve the non-negotiable identity or message

Avoid using prompt quality as proof of output quality.

## Operations and workflows

- One real or safe test request completes from intake to confirmation
- Owners and handoffs are observable
- Failure recovery works in a controlled test
- Required logs, approvals, or records exist

Avoid treating a written procedure as proof that the process works.

## Data and spreadsheets

- Known examples produce known results
- Totals reconcile to an independent source
- Formulas are inspected and recalculated
- Required ranges, types, and constraints validate

Avoid calculating expected values with the same formula being tested.

## Communications

- Recipient, intent, facts, tone, and requested action are correct
- The draft is reviewed before sending
- Links and attachments are present when required

Draft approval is proof of readiness, not authorization to send.

## Personal plans

- Calendar, budget, energy, travel, or dependency constraints are checked
- The first action fits the user's real schedule
- Progress can be observed without vague self-ratings

Avoid plans that prove only that a checklist was created.

## Software and systems

- Automated behavior test through a public interface
- Browser or device check for user-visible behavior
- Build, lint, schema, security, or deployment checks where they match the risk
- Regression proof that fails without the change and passes with it

Avoid tests tied only to internal calls or mocks of owned code.
