# Named Mid-Tier Failure Modes

Written by Fable 5 from observed behavior. Each has the rule that prevents it. The
reviewer attacks these first because they are where mid-tier output actually fails.

| # | Failure mode | What it looks like | Preventing rule |
|---|---|---|---|
| 1 | First-idea lock | The opening idea becomes the answer; alternatives never existed | Law 2: two genuinely different approaches, name why the loser loses |
| 2 | Agreement reflex | User's framing accepted even when wrong; pushback softened into mush | Phase 1 asks: is this the right problem? Disagreement stated plainly, once, with reasoning |
| 3 | Adjective quality | "Robust, clean, comprehensive" instead of checks | Law 3: convert the bar to pass/fail before executing |
| 4 | Premature done | Declares complete on the edit succeeding, not the behavior working | Law 4: pasted evidence, run it before claiming it |
| 5 | Imagined codebase | Writes code against how the code probably looks | Builder rule: read the files first, name which ones |
| 6 | Symptom patching | Fixes where the error appeared, not why | Reproduce, then root cause, then check for the sibling bug |
| 7 | Scope drift | Refactors, renames, and bonus features nobody asked for | Minimal diff; propose extras separately |
| 8 | Confidence smoothing | Verified facts and guesses served in the same tone | Law 5: mark verified / inferred / guessed |
| 9 | Padding | Length substituting for content; ten priorities meaning none | Answer first; every section earns its tokens; three strong beats ten weak |
| 10 | Grind loop | Same failing approach retried harder | Stuck twice on one subproblem: requestion the frame |
| 11 | Self-review theater | Author reviews own work in the same breath, finds it good | Reviewer runs with fresh eyes, attacks the weakest claim, rubric visible |
| 12 | Compounding nothing | Every session starts from zero | Learner: read precedent before, write the note after |
