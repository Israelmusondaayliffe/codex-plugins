# Matt To Spec

Synthesize what is already known. Do not restart the interview. If consequential decisions remain unresolved, list them as blockers rather than hiding them as assumptions.

## Process

1. Read the current conversation, project instructions, existing terminology, decisions, prototypes, and research artifacts.
2. Identify the observable outcome and its real verification surface.
3. For software, record affected modules, interfaces, seams, behavior, migration needs, operational risks, and testing decisions without brittle file-by-file prescriptions.
4. For knowledge work, record audience, decision or use case, evidence requirements, deliverable shape, review criteria, and delivery constraints.
5. Write the spec to the configured local tracker or user-approved durable path.
6. Publish to an external tracker only when the task authorizes it, then verify the created item.

## Template

### Problem or opportunity

Describe the current state and why it matters from the user's perspective.

### Desired outcome

Describe what will be observably true when the work succeeds.

### Users and uses

List the actors, decisions, or workflows the result must support.

### Scope and requirements

Number the required behaviors or deliverable properties. Keep them independently checkable.

### Decisions already made

Record settled product, implementation, editorial, analytical, operational, and evidence choices.

### Verification plan

Name the checks, review axes, sources, renderers, tests, or live surfaces that prove completion.

### Out of scope

State exclusions clearly enough to prevent accidental expansion.

### Open blockers

List only unresolved questions that prevent safe execution.
